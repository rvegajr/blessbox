# Fix: Security Headers + Strict TS Build

Date: 2026-04-25
Author: rvegajr@noctusoft.com

## Summary

Added a `headers()` block to the Next.js config delivering HSTS, X-Content-Type-Options, X-Frame-Options, Referrer-Policy, Permissions-Policy, and a CSP for all routes. Also removed `typescript.ignoreBuildErrors: true`. Build passes clean and headers verified at runtime.

## Critical finding: dead config file

`src/next.config.js` (the file the QA report flagged) was **never being loaded by Next.js**. Next looks for `next.config.{js,mjs,ts}` in the project root — the `src/` location was a no-op. The previous "config" (image remotePatterns, server externals, ignoreBuildErrors, etc.) was effectively unused; Next was running on defaults the entire time.

Fix: created `/Users/admin/Dev/YOLOProjects/BlessBox/next.config.js` at the project root with the full intended config + `headers()`, and `git rm`'d the dead `src/next.config.js`.

## Inline scripts on /checkout (CSP-allowed via 'unsafe-inline')

`/checkout` ships Next.js Flight payload bootstrap scripts: `<script>(self.__next_f=self.__next_f||[]).push(...)</script>` (multiple). These are inline, generated by Next, and require `'unsafe-inline'` on `script-src`. No third-party inline scripts observed.

## Follow-ups (not done)

- `'unsafe-inline'` on `script-src` is required by Next.js 16 App Router Flight payloads. Tightening requires nonces (`next.js` middleware + `headers()` callback returning per-request nonces) — non-trivial, recommend separate ticket.
- `wss:` in `connect-src` is broad; once the Turso websocket host is confirmed (e.g. `wss://*.turso.io`), narrow it.
- The Square sandbox host should only be allowed in non-production builds; consider gating via `process.env.NODE_ENV` in `next.config.js`.
