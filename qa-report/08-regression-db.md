## Section 17 — Database & Migrations

### 17.1 Migrations directory & `drizzle-kit check`

- `drizzle.config.ts` (at `src/drizzle.config.ts`) declares `out: './drizzle/migrations'`.
- **No `drizzle/migrations/` directory exists** in the repo (no `.sql` migration files anywhere outside `node_modules`). The project uses `db:migrate` = `drizzle-kit push` (push-style, schema-driven), not generated SQL migrations.
- `npx drizzle-kit check --config=src/drizzle.config.ts` output:
  ```
  Reading config file '/Users/admin/Dev/YOLOProjects/BlessBox/src/drizzle.config.ts'
  Everything's fine 🐶🔥
  ```
  No drift reported (also: nothing to compare since migrations dir is empty).

**Finding (advisory, not regression):** Production schema management relies on `drizzle-kit push` + ad-hoc scripts (`scripts/run-migration.js`, `scripts/create-turso-schema.js`, `scripts/update-turso-schema.js`, `scripts/verify-production-schema.ts`). There is no versioned SQL migration history. Acceptable for a small team but risky for rollback / audit.

### 17.2 Schema vs DB spot-check (3 tables)

Schema in `lib/schema.ts` (single file, 219 lines, 17 tables). Local DB (`blessbox.db`) actually contains **24 tables** including some not in `lib/schema.ts` (e.g. `export_jobs`, `saved_searches`, `usage_alerts`, `usage_tracking`, `payment_transactions`, `login_codes`, `coupon_codes`). These were created by ad-hoc scripts and are NOT in the Drizzle schema — schema drift.

Spot-check:

| Table | `lib/schema.ts` | `blessbox.db` | Match? |
|---|---|---|---|
| `verification_codes` | id, email, code, attempts, created_at, expires_at, verified | identical columns/types/defaults | YES |
| `organizations` | id, name, eventName, customDomain, contact_email/phone/address/city/state/zip, passwordHash, lastLoginAt, emailVerified, created/updated_at | matches (camelCase mapped to snake_case via column names) | YES |
| `registrations` | id, qrCodeSetId, qrCodeId, registrationData, ipAddress, userAgent, referrer, deliveryStatus, deliveredAt, registeredAt, checkInToken, checkedInAt, checkedInBy, tokenStatus, organizationId | DB matches; `organization_id` was added via ALTER (visible mid-line in schema dump) | YES |

Core 3 tables match. But `users` table in DB has extra columns (`name`, `image`, `email_verified_at`) added via ALTER that are NOT in `lib/schema.ts` (which defines no such columns). **Schema drift** in `users` and several auxiliary tables.

### 17.3 Backup / restore strategy

`scripts/` audit for `blessbox.db` backup/restore: **none found**.

- No `scripts/backup*.sh`, no `restore*.sh`, no `dump*.{sh,js,ts}`, no cron/CI artifact.
- Closest related: `scripts/clear-production-database.ts`, `scripts/clear-production-database-interactive.ts`, `scripts/run-clear-database.sh` — these DESTROY data, no backup taken first.
- `scripts/verify-production-schema.ts` and `scripts/check-production-data.ts` are read-only checks.

**Gap:** no documented backup/restore script for `blessbox.db` (local) or for Turso prod (Turso has its own snapshots, but no repo-side script invokes them).

## Summary

- **Section 16:** All audited fixes (Square v43 `token`, SendGrid lazy-init, verification email failure handling, plus 7 other historical fixes) are present and intact. **Zero regressions.**
- **Section 17:** `drizzle-kit check` clean, but the project has **no generated SQL migrations** (push-only) and **schema drift** between `lib/schema.ts` (17 tables) and the live DB (24 tables, plus extra columns on `users`). **No backup/restore scripts** for `blessbox.db`. Core tables match schema; auxiliary tables are managed outside Drizzle.

### Recommendations

1. Generate baseline SQL migrations: `npx drizzle-kit generate` and commit to `drizzle/migrations/`.
2. Reconcile drift — add `users.name/image/emailVerifiedAt` and the 7 missing tables to `lib/schema.ts`, or remove them from the DB if unused.
3. Add `scripts/backup-db.sh` (sqlite3 `.backup` for local; Turso CLI `db shell ... .dump` for prod) and run before any `clear-production-database*` script.
