## 5. QR Codes

| Endpoint | Method | Status | Notes |
|---|---|---|---|
| `/api/qr-codes` | GET | 401 | Auth gated |
| `/api/qr-codes` | POST | 405 | No POST handler — see bug below |

### Authz model (source)
All `/api/qr-codes*` and `/api/qr-code-sets` routes call `getServerSession()` then `resolveOrganizationForSession()`, returning 401/409 respectively. For per-QR routes (`[id]`, `download`, `analytics`), the route fetches the parent set via `qrCodeService.getQRCodeSet(qrCodeSetId)` and rejects with 403 unless `qrCodeSet.organizationId === activeOrg.id` or the caller is `super_admin`. Org scoping is correct.

`/api/onboarding/generate-qr` uses `MembershipService.isMember(userId, organizationId)` → 403 if not a member. Good.

`/api/qr/download` is **public** (no auth). It only zips client-supplied `dataUrl` PNGs into a ZIP/PDF — no DB access, no PII leak. Acceptable as a stateless utility, but worth flagging because it differs from sibling routes.

### QR data format / URL
From `app/api/onboarding/generate-qr/route.ts:185`, the encoded URL is:

```
${baseUrl}/register/${orgSlug}/${entryPoint.slug}
```

`baseUrl` resolves from `PUBLIC_APP_URL` → `NEXTAUTH_URL` → `NEXT_PUBLIC_APP_URL` → Vercel URL → `http://localhost:7777`. PNG generated by `qrcode` package, `errorCorrectionLevel: 'M'`, 256px, brand color `#667eea`.

`/api/qr-codes/[id]/download` returns `Content-Type: image/png` and `Content-Disposition: attachment; filename="qr-code-<id>.png"` from a base64 data URL stored in the QR record. Could not verify non-empty body without auth (needs human with session).

### Bugs / concerns

1. **No POST/Create handlers on `/api/qr-codes` or `/api/qr-code-sets`.** Both return 405. Header comment in `qr-codes/route.ts` line 9 says `POST (create)` but the function is missing. UI must be using `/api/onboarding/generate-qr` exclusively. Either remove the misleading comment or add the handler.
2. `PUT /api/qr-codes/[id]` (line 113-117) has comment "label/slug/url/dataUrl are intentionally immutable here" but still accepts a `label` field and forwards it as `description`. Confusing; rename or drop.
3. Real phone scan verification: **needs human.** Cannot scan; URL format above can be eyeballed by decoding the encoded string.

## 6. Check-in

### Endpoints

| Endpoint | Method | Status (unauth) | Notes |
|---|---|---|---|
| `/api/check-in/search?q=test` | GET | 401 | Auth gated |
| `/api/registrations/[id]/check-in` | POST | 404 (bad id) | Allows token OR session auth |
| `/api/registrations/[id]/undo-check-in` | POST | 404 (bad id) | Allows token OR session auth |

### Authz / org scoping (source)

`/api/check-in/search/route.ts:42-55` builds SQL `JOIN qr_code_sets qcs ON r.qr_code_set_id = qcs.id WHERE qcs.organization_id = ?` with `args: [organization.id]`. **Cross-org leakage check: PASS** — registrations are filtered by the active org's QR sets at the SQL level.

`/api/registrations/[id]/check-in` accepts EITHER a matching `checkInToken` from the registration OR a session that owns the QR code set. Token path bypasses org scoping by design (the QR token grants self-service check-in). Session path verifies ownership via `qr_code_sets.organization_id`.

### Bugs / concerns

1. **Race condition on double check-in.** `RegistrationService.checkInRegistration()` lines 489-525 does `SELECT checked_in_at` then `UPDATE` in two separate statements with no transaction and no `WHERE checked_in_at IS NULL` guard. Two concurrent calls can both pass the SELECT and both UPDATE, producing duplicate "successful" check-ins (last writer wins on `checked_in_by`). **Fix:** make the UPDATE conditional (`WHERE id = ? AND checked_in_at IS NULL`) and treat `rowsAffected === 0` as "already checked in". App-level guard works for sequential calls (returns 409 with "already checked in" message — verified in error mapping at route line 85-90), but not for concurrent.
2. **Suspicious indentation/brace nesting** in both `check-in/route.ts` (lines 45-67) and `undo-check-in/route.ts` (lines 31-47). The `db.execute` org-ownership check is inside the `if (!organization)` block's else implicitly via brace placement — re-reading carefully, the block IS structurally inside `if (isSessionAuth && !isTokenAuth)`, so behavior is correct, but the indentation is misleading and a refactor risk. Worth cleaning up.
3. Token-based check-in path does not verify the registration's QR set is still active or that the org subscription is current. Low priority.

## 7. Participants / Classes / Enrollments

### Endpoints

| Endpoint | Method | Status (unauth) | Notes |
|---|---|---|---|
| `/api/participants` | GET/POST | 401 | Auth gated |
| `/api/classes` | GET/POST | 401 | Auth gated |
| `/api/classes/[id]` | GET | 401 | **No PUT/DELETE handlers exist** |
| `/api/classes/[id]/sessions` | GET/POST | 401 | Auth gated |
| `/api/enrollments` | GET (needs `classId`)/POST | 401 | Auth gated |

### Authz / org scoping (source)

All routes use `getServerSession` + `resolveOrganizationForSession` and `ensureDbReady`. List endpoints scope by `organization.id` via the service layer. `/api/classes/[id]` GET, sessions GET/POST, enrollments GET/POST all explicitly check `cls.organization_id !== organization.id → 403`. Org isolation looks correct.

### Bugs / concerns

1. **Missing PUT/DELETE on `/api/classes/[id]`.** Only GET implemented. Editing or removing a class via UI will fail. High-priority gap.
2. **Capacity race condition in enrollment.** `app/api/enrollments/route.ts:75-90`: code does `getEnrollmentsByClass(class_id)`, counts non-cancelled, compares to `classDetails.capacity`, then `enrollParticipant()` — three round trips, no transaction, no DB-level uniqueness/capacity constraint. Concurrent POSTs can both pass the capacity check and both insert, exceeding capacity. **Fix:** wrap in transaction with `SELECT ... FOR UPDATE` (or sqlite equivalent of an atomic INSERT guarded by a count CTE), or use a unique-slot table.
3. **No waitlist logic.** Spec/task mentions "capacity/waitlist logic"; source only enforces a hard 409 "Class capacity reached". Waitlist appears unimplemented.
4. **Email-send failure swallowed silently** (line 116-118) — by design (dev), but the response still returns 201 with no indication that invitation email was skipped. Consider returning a `warnings` field.
5. POST `/api/participants` does not check for duplicate email within the org. Could result in duplicates.
6. POST `/api/classes` accepts `capacity` as string and parses with `parseInt`; no validation that it is positive. `capacity: 0` short-circuits the capacity check (`capacity > 0 && ...`) → unlimited enrollment.
7. Session POST has no validation of `session_date`/`session_time` format and no class-window/timezone sanity.

---

## Summary of high-impact issues

- **Check-in race**: non-atomic check-in update permits double check-ins under concurrency. (Service layer.)
- **Enrollment race**: non-atomic capacity check permits over-enrollment.
- **Missing PUT/DELETE for `/api/classes/[id]`** — UI edit/delete will not function.
- **No waitlist implementation** despite product expectation.
- **POST handlers missing** on `/api/qr-codes` and `/api/qr-code-sets` despite header comments.
- **`capacity: 0` means unlimited** — undocumented surprise.
