# Fix: Zod Input Validation

Closes the gap from `qa-report/07-security.md` (no schema-validation library)
and the request-size concern from `qa-report/02-orgs-registrations.md`.

## What changed

- Promoted `zod` to a direct `dependencies` entry in `package.json` (was only
  reachable transitively via `traklet`).
- Added `lib/api/validate.ts` with:
  - `parseBody<T>(req, schema, opts?)` returning `{data}` on success or
    `{error: NextResponse}` (status 400 with `{error: 'Validation failed',
    details: zodError.flatten()}`).
  - Hard 1MB body cap (`DEFAULT_MAX_BODY_BYTES`) enforced **before**
    `JSON.parse` via Content-Length precheck + streamed read with abort.
    Oversize → 413.
  - Malformed JSON → 400 `{error: 'Invalid JSON body'}`. No stack traces,
    no parser internals, no engine details.
  - Tolerant of NextRequest mocks (falls back to `.text()` / `.json()` when
    `req.body` is not a ReadableStream — keeps existing unit tests green).
  - `parseValue<T>(value, schema)` companion for query-string params.
- Tests: `lib/api/validate.test.ts` (8 cases) covering valid input, malformed
  JSON, schema mismatch, oversize body (Content-Length + streamed), empty
  body, and `parseValue`. Verifies no `stack` / `SyntaxError` /
  `Unexpected token` strings leak in error responses.

## Routes hardened (12)

Each gets a zod schema + `parseBody`:

| Route | Schema highlights |
|---|---|
| `app/api/auth/send-code/route.ts` | `email: z.string().email()` |
| `app/api/auth/verify-code/route.ts` | `email`, `code: /^\d{6}$/`, optional `organizationId` |
| `app/api/onboarding/send-verification/route.ts` | `email` |
| `app/api/onboarding/verify-code/route.ts` | `email`, `code: /^\d{6}$/`, optional `organizationId` |
| `app/api/onboarding/create-organization/route.ts` | `name`, `contactEmail`, optional address fields with length caps |
| `app/api/onboarding/save-organization/route.ts` | Same as above (email comes from session) |
| `app/api/onboarding/save-form-config/route.ts` | `organizationId`, `formFields: array(unknown).max(200)`, optional `language`/`name`/`autoGenerateDefaultQR` |
| `app/api/onboarding/generate-qr/route.ts` | `organizationId` + permissive entry-point shapes (preserves `formConfigId`/`qrCodes`/`qrLabel` backwards-compat) |
| `app/api/payment/process/route.ts` | `planType`, `billingCycle`, optional `paymentToken`/`nonce`/`couponCode`. **Deliberately omits `amount`** — server-derived from `getPlanPriceCents()`. |
| `app/api/registrations/route.ts` POST | `orgSlug`, `qrLabel`, `formData: record(string, unknown)`, `metadata` |
| `app/api/registrations/submit/route.ts` | Same |
| `app/api/report-bug/route.ts` | `description`/`expected`/`steps` (1–5000), `browser`/`device` required, `images` capped at 10, `labels` capped at 10. Body cap raised to 10MB to allow base64 screenshots. |
| `app/api/coupons/validate/route.ts` | `code: 1–50` |
| `app/api/coupons/apply/route.ts` | `code`, `planType`, `amountCents: int >= 0` |
| `app/api/check-in/search/route.ts` | Query params: `q ≤ 200`, `filter ∈ {all,pending,checked-in}`, `limit ∈ [1,500]` (coerced from string) |

## Verification

- `npm run build` — passes; no type errors introduced.
- `npm run test -- --run` — **373/373 pass** (32 test files), including the
  new validate suite and the existing PaymentProcess tests (which mock
  `req.json()`; the helper transparently handles both real Requests and
  these mocks).

## Notes / non-goals

- The legacy `safeParseJson` in `lib/api/errorResponse.ts` is left in place
  for the many other routes not in scope; new routes should prefer
  `parseBody`.
- Bug-report route keeps the original `BugReportPayload` interface but the
  zod schema is now the source of truth; `as BugReportPayload` cast is a
  shim until the type is removed.
- `payment/process` still accepts `nonce` as an alias for `paymentToken`
  (matches Square Web SDK terminology); both go through the same validated
  path.
