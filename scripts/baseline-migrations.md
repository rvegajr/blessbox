# Baseline Migrations

The first migration in `drizzle/migrations/` (`0000_baseline.sql`) is a full
`CREATE TABLE ... IF NOT EXISTS` snapshot of the schema as of 2026-04-25,
generated by:

```bash
npx drizzle-kit generate --config=src/drizzle.config.ts --name=baseline
```

It mirrors the live `blessbox.db` (and the Turso production DB) exactly, so it
must be marked as **already applied** in any environment that already has the
schema. Otherwise `drizzle-kit migrate` (or any runner that consumes
`drizzle/migrations/_journal.json`) will try to recreate tables.

## Mark baseline as applied (production / existing envs)

Drizzle stores applied-migration state in a `__drizzle_migrations` table
(created on first run). To pre-seed it for an environment that is already at the
baseline schema:

1. Compute the SHA-256 hash of `0000_baseline.sql`:

   ```bash
   shasum -a 256 drizzle/migrations/0000_baseline.sql | awk '{print $1}'
   ```

2. Read the migration tag from `drizzle/migrations/meta/_journal.json` — the
   `tag` field of the entry whose `idx` is `0` (e.g. `"0000_baseline"`).

3. Connect to the target DB (Turso CLI or `sqlite3` for local) and run:

   ```sql
   CREATE TABLE IF NOT EXISTS __drizzle_migrations (
     id INTEGER PRIMARY KEY AUTOINCREMENT,
     hash TEXT NOT NULL,
     created_at NUMERIC
   );

   INSERT INTO __drizzle_migrations (hash, created_at)
   VALUES ('<sha256-from-step-1>', strftime('%s','now') * 1000);
   ```

   Drizzle's libSQL/SQLite migration runner identifies applied migrations by
   the SHA-256 hash of the SQL file contents stored in this table.

4. Verify:

   ```bash
   sqlite3 blessbox.db "SELECT * FROM __drizzle_migrations;"
   ```

5. Subsequent `drizzle-kit generate` runs (after future schema edits) will emit
   `0001_*.sql`, `0002_*.sql`, … and the runner will apply only those — the
   baseline will be skipped because its hash is already recorded.

## Fresh local environments

For a brand-new local DB (no existing tables), do **not** pre-seed
`__drizzle_migrations`. Just run the migration runner of choice; the baseline
will create everything from scratch and record itself as applied.

## Notes

- All `CREATE TABLE` statements in `0000_baseline.sql` use `IF NOT EXISTS`
  semantics where Drizzle generates them, so re-applying against a live DB is
  generally safe but still wasteful — pre-seeding is the correct path.
- Indexes also use `IF NOT EXISTS` in Drizzle's generated SQL.
- Always run `scripts/backup-db.sh` before applying any migration to a DB that
  contains real data.
